# Query 理解与改写

## 第 5 章 Query 理解与改写

用户输入的 query 往往很短，也不一定规范。搜索链路首先要把这串字符整理成可供召回和排序使用的结构化信号：

```text
raw query
  -> 规范化 / 分词 / 实体识别
  -> 词权重 / 类目 / 意图
  -> 原词与改写词
  -> 召回路由、过滤条件和排序特征
```

Query 理解一旦出错，后面的召回和排序都会沿着错误的结构化信号工作。

### 5.1 分词要确定检索边界

中文没有天然空格。基于词典的方法会在字符序列上寻找合法词，并用动态规划选择代价最小的切分：

```math
\operatorname{cost}(0,t)
=\min_{s<t}
\left[
\operatorname{cost}(0,s)
-\log P(x_{s:t})
\right].
```

词频高的切分代价低。工程上还会给品牌、型号、人名、地点和业务词加权，避免把 `"iPhone 15 Pro Max"` 或 `"三体电视剧"` 切坏。

词典永远追不上新词。构造词典时通常结合：

- 站内 query 和文档中的词频；
- 左右邻接熵，判断候选片段的边界是否自由；
- 点互信息，判断片段内部字符是否经常一起出现；
- 人工维护的实体库、品牌库和敏感词库。

深度学习分词把任务写成 BMES 序列标注。BERT 给每个字符编码，分类头或 CRF 预测 `B/M/E/S` 标签。它更能利用上下文，但词典仍适合处理刚出现的商品型号、强规则实体和必须可控的边界。

### 5.2 命名实体识别

NER 要识别人名、地点、品牌、作品、机构、商品型号和 POI 等实体。例如：

```text
"苹果 15 北京维修"
品牌=苹果
型号=iPhone 15
地点=北京
意图=维修服务
```

实体结果会进入多个下游：

- 选择垂类索引；
- 做实体规范化和别名扩展；
- 添加品牌、地点或型号过滤；
- 从知识库、商品库或 POI 库召回；
- 构造 query-document 交叉特征。

NER 的评价应按完整实体 span 计算 precision、recall 和 F1。只算字符级准确率会掩盖边界错误；`"北京大学"` 被识别成 `"北京"`，字符大多预测正确，业务含义却已经变了。

### 5.3 词权重

Query 中的词并不等价。`"苹果 手机 壁纸"` 里，删除 `"手机"` 可能改变类目，删除 `"壁纸"` 会改变核心需求，删除某些修饰词的影响则较小。

一种标注方法是逐词遮盖，观察人工标注者认为 query 意图改变了多少。模型可在 BERT 输出上预测每个 term 的权重：

```math
\alpha_j
=\frac{\exp(w^\top h_j)}
{\sum_k \exp(w^\top h_k)}.
```

词权重可用于：

- 倒排召回中的 term boost；
- 改写时保留核心词；
- SUG 的核心词召回；
- 相关性特征；
- query 类目和意图识别。

不要直接拿 IDF 当语义词权重。IDF 反映词在文档集合中的稀有程度，业务词权重反映它对当前 query 意图的重要程度。二者有时相关，但不是同一个量。

### 5.4 类目识别

一个 query 可以属于多个类目。`"儿童防晒衣"` 同时涉及童装、防晒和户外，适合多标签分类：

```math
p_c=\sigma(w_c^\top h_q), \qquad c=1,\ldots,C.
```

线上不能简单地对所有类目使用同一阈值。头部类目数据多、先验概率高，长尾类目需要单独校准。评价时同时看 micro-F1、macro-F1 和分层召回：

- micro-F1 更受头部类目影响；
- macro-F1 能暴露长尾类目是否完全失效；
- top-k recall 反映候选索引是否包含正确类目。

类目预测用于召回路由和排序特征，不宜过早变成硬过滤。低置信度类目一旦硬过滤，相关文档连后续纠错机会都没有。

### 5.5 意图识别

意图标签的价值在于改变下游调用，而不是给 query 贴一个好看的名字。常见意图包括：

| 意图 | 可能触发的链路 |
| --- | --- |
| 商品购买 | 商品索引、价格和库存过滤 |
| 本地服务 | POI 识别、距离召回和地域排序 |
| 突发事件 | 热点索引、时效性提权 |
| 医疗或金融 | 高质量来源、安全护栏 |
| 找人或找机构 | 实体库、账号库 |
| 明确导航 | 直达卡片或高精度规则 |

多意图 query 不应被强行压成一个标签。模型可以输出多标签概率，再由路由层按置信度并行调用多条通道。

### 5.6 Query 改写

改写的目的，是让原词找不到的相关内容进入候选集。常见改写包括：

- 错别字与拼音纠错；
- 简繁体、大小写和单位规范化；
- 别名、缩写和同义词扩展；
- 分词修正；
- 实体链接与型号标准化；
- 省略补全；
- 根据意图做扩展或收缩。

有三类常见做法。

第一类基于分词和词典。它便宜、可解释，适合拼写、别名和核心词扩展。

第二类基于相关性。候选改写词先召回，再用 query-query 或 query-document 相关性模型筛掉语义漂移。

第三类基于意图。模型根据上下文生成改写，但必须保留核心实体和约束。例如 `"苹果手机维修"` 不能因为用户常看食谱而改成水果相关内容。

线上通常保留原 query 通道：

```text
原 query 召回 ─┐
高置信改写召回 ├─> 合并、去重、标记 rewrite_source
探索性改写召回 ┘
```

每个候选要记录来自哪个改写。否则出现坏结果时，只能看到最终 query，找不到是哪条 rewrite 把语义带偏。

### 5.7 怎样评价 Query 理解

组件指标只能回答模型是否完成自己的任务：

- 分词与 NER：span precision / recall / F1；
- 类目与意图：分层 precision、recall、micro/macro-F1；
- 改写：人工相关率、覆盖率、改写撤回率；
- 路由：正确索引覆盖率、空召回率。

还要沿链路观察：

- 无结果率是否下降；
- 召回正例是否增加；
- 相关性是否下降；
- 用户主动换词率是否下降；
- 长尾 query 是否真正改善。

改写带来更多候选，却让不相关结果变多，不算成功。

<summary>参考答案</summary>

1. 高频字符片段可能只是固定搭配的一部分。还要看片段内部凝聚度和左右边界自由度，并结合实体库与人工审核。
2. 业务需要完整实体。边界少一个字就可能链接到不同品牌、地点或机构；字符准确率会把这种严重错误稀释掉。
3. IDF 衡量词在文档集合中的稀有程度；query 词权重衡量词对当前搜索意图的重要程度。
4. 一个 query 可以同时命中多个类目和属性。强行单分类会丢掉并行召回所需的信号。
5. 保留原 query 兜底，限制核心实体和高权重词，给改写打置信度，并用相关性模型和人工切片评估过滤。
6. 标签应控制索引选择、召回通道、过滤、排序特征或安全策略。只离线输出一个分类结果而不改变下游，业务上没有完成闭环。
