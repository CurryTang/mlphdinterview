# 第六部分：把知识用到题目里

## 第 15 章 三个系统设计案例

系统设计题没有唯一架构，但有一个可靠顺序：

```text
目标与约束
  -> 数据和标签
  -> 候选生成
  -> 排序与列表决策
  -> 在线服务
  -> 评价与实验
  -> 风险和降级
```

一上来就画双塔或 Transformer，通常会漏掉决定方案的业务条件。

### 案例一：短视频推荐

#### 目标

假设目标是提升用户长期有效消费，同时控制负反馈和内容同质化。不要只写 CTR。

主指标可以是有效观看时长或次日留存，中间指标包括点击、完播、点赞和关注，护栏包括退出、举报、夜间过度消费、作者集中度与 P99 延迟。

#### 数据

正反馈按强度区分：有效播放、完播、点赞、关注、分享。快速划走、点不感兴趣和举报是负反馈。曝光日志必须带位置、召回通道、模型版本和规则原因。

序列样本按时间切，严禁读取标签发生后的行为。转化较慢的标签要设置成熟窗口。

#### 召回

可以并行使用：

- ItemCF/Swing；
- 用户-物品双塔；
- 关注作者；
- 内容相似；
- 热门与地域；
- 冷启动池；
- 生成式召回实验通道。

各路记录来源并去重。生成式召回若使用 Semantic ID，先作为补充通道，便于测独立增量和无效解码率。

#### 排序

粗排用小模型保住精排 top 候选。精排做多目标预估，可用 MMoE 共享点击、时长、点赞等任务。时长要按视频长度校准，负反馈单独建模。

行为序列用 Last-N + DIN 起步。序列很长时再考虑 SIM 或 HSTU 类结构，先确认实时行为更新和时延预算。

#### 重排

使用 MMR 或规则化贪心控制作者、类目和内容相似度；加入冷启动配额、广告频控、安全过滤。探索内容必须有预算和质量底线。

#### 服务

物品 embedding 和大部分静态特征离线计算，用户短期序列流式更新。ANN、特征服务和排序模型都要有版本。降级顺序可以是：

```text
复杂精排失败 -> 小精排
向量召回失败 -> ItemCF/热门
实时特征失败 -> 最近快照
重排失败 -> 安全规则 + 精排顺序
```

#### 实验

先看召回覆盖、精排 NDCG/校准和重排列表指标，再做 A/B。新品探索要单独分层，避免主实验把探索收益与模型收益混在一起。长期 Holdout 观察留存和内容生态。

### 案例二：电商搜索

#### 目标

用户输入 query 后尽快找到能买的商品。相关性是底线，成交额不是唯一目标。若把高利润但不相关商品推上来，短期收入可能涨，搜索信任会掉。

主指标可选搜索成功率、购买转化或 GMV，过程指标有无结果率、改写率、点击和加购，护栏包括相关性人工分、退货、投诉、延迟和库存错误。

#### Query 理解

处理分词、品牌/型号实体、类目、价格与属性约束、拼写纠错和改写。

例：

```text
"苹果 15 256 黑色"
 -> brand=Apple
 -> model=iPhone 15
 -> storage=256GB
 -> color=black
```

强结构属性应进入过滤或结构化召回，不应只依赖 embedding。

#### 召回

- 倒排/BM25 保证精确 term；
- 属性过滤保证品牌、型号、库存；
- query-item 双塔补语义；
- query 改写多路召回；
- 高频 query 的缓存结果；
- 商业或新品通道在明确配额内进入。

Hybrid 结果可用 RRF 或学习融合，保留每路归因。

#### 排序

前级先做相关性，Cross-BERT 或轻量 cross-encoder 处理 top-N。随后融合质量、销量、价格、库存、时效、个性化和商业目标。

比较稳的结构是：

```text
相关性底线
  -> 质量/可购买性
  -> 个性化与业务目标
  -> 列表规则
```

不要让 GMV 预估直接把弱相关商品顶到第一位。

#### 生成式能力放哪里

LLM 可用于复杂 query 解析、属性抽取、改写、top-20 重排或商品问答。常规导航 query 不需要 agentic search。

如果页面提供"直接回答"，例如比较两款相机，应把商品结构化数据、详情和可信评测作为证据，答案中的价格和库存必须实时查询，不能依赖模型记忆。

#### 评价

离线分开评 query 理解、召回 Recall、相关性 NDCG、属性约束准确率。线上看搜索成功、成交和长期回访。按头部/尾部 query、品牌词、类目词和复杂属性词分桶。

### 案例三：带引用的生成式搜索

#### 目标与边界

这个系统要给出正确、完整、可追溯的回答。"写得像答案"远远不够，系统还必须能说不知道。

先定义数据范围：公开网页、企业内部文档或特定知识库。权限、时效和来源质量会直接改变架构。

#### 建库

1. 抓取或接收文档；
2. 清洗正文，保留标题、时间、作者、URL 和权限；
3. 按结构切块，建立父子关系；
4. 建 BM25 与向量索引；
5. 记录版本和删除标记。

金融、法律、医疗等高风险场景需要更严格的来源白名单与更新 SLA。

#### 在线

```text
问题分类
  -> 是否需要搜索
  -> query rewrite / decomposition
  -> hybrid retrieval
  -> rerank
  -> 上下文去重与压缩
  -> 基于证据生成
  -> claim-citation 对齐
  -> 事实核验与拒答
```

简单事实可单轮处理；多跳问题进入有最大步数和费用预算的搜索循环。每次搜索保留 query、返回文档、选择理由和时间。

#### 生成约束

提示模型只使用给定来源不够。还需要：

- 将引用绑定到具体 chunk；
- 对答案拆 claim；
- 检查 claim 是否被引用段落蕴含；
- 冲突来源不强行合并；
- 证据不足时降级为搜索结果列表或拒答。

#### 评价

建立问题集时同时标注答案与证据。按以下漏斗诊断：

```text
证据是否在库里？
  -> 是否被召回？
  -> 是否进入最终上下文？
  -> 模型是否使用？
  -> 引用是否真的支持 claim？
```

离线还要测 prompt injection、恶意网页、越权文档和过期来源。线上看用户追问、引用点击、纠错和放弃率，不能只用 LLM-as-judge 给一个总分。

### 15.1 如何回答开放式"涨指标"

先定位漏斗，不要先报模型名。

召回问题：

- 正例不在候选；
- 新兴趣或长尾覆盖不足；
- 过滤/配额误杀；
- 索引更新慢。

排序问题：

- 标签与目标错位；
- 样本偏差；
- 特征缺失或穿越；
- 多目标融合和校准错误；
- 粗排误杀。

列表问题：

- 重复和疲劳；
- 规则过强；
- 冷启动没有机会；
- 短期收益伤害长期体验。

工程问题：

- P99 延迟；
- 特征版本；
- 缓存陈旧；
- 模型或索引灰度不一致；
- 降级比例上升。

诊断出层级后再选择数据、模型、策略或工程优化。模型升级只是其中一类动作。

### 15.2 上线前后怎样验证

组件做完后再谈指标，先明确它对应哪一层：

| 层级 | 例子 | 用途 |
| --- | --- | --- |
| 组件指标 | Recall、NDCG、LogLoss、校准误差、P99 | 判断召回、排序或服务是否按预期工作 |
| 过程指标 | CTR、有效播放、加购、无结果率 | 解释用户行为在哪一步变化 |
| 业务结果 | 留存、总时长、成交、搜索成功 | 决定产品是否获得真实收益 |
| 护栏 | 投诉、安全、延迟、作者集中度、商业化负载 | 防止用有害方式换主指标 |

离线结果只能决定是否值得进线上实验。常见的离线线上落差包括：候选集不同、标签与业务目标错位、特征穿越、规则抵消模型收益，以及延迟增加。

A/B 测试使用稳定分桶：

```text
bucket = hash(user_id, experiment_salt) mod B
```

同一用户应稳定落桶。相互冲突的实验在同层互斥，可并行的实验放在正交层。实验前先做 AA 检查，并提前确定随机化单元、主指标、护栏、最小可检测效应和观察窗口。

结果不能只看 `p < 0.05`。同时报告差值、置信区间和样本量；若置信区间仍包含业务上有意义的收益，结论应是证据不足，而非策略无效。

长期 Holdout 用于观察多次发布的累计效果。推全后还可保留小流量反转桶，确认收益没有被流量变化、新奇效应或同期活动误导。

#### Quick Coding：稳定 A/B 分桶

用 SHA-256 实现稳定分桶，不能使用 Python 内置 `hash()`。除了确定性和取值范围，还要处理非法桶数，并解释不同实验为什么需要不同 salt。

## 公式速查

### 召回与相似度

余弦相似度：

```math
\cos(a,b)=\frac{a^\top b}{\|a\|_2\|b\|_2}.
```

ItemCF：

```math
\operatorname{sim}(i,j)
=\frac{|U_i\cap U_j|}
{\sqrt{|U_i||U_j|}}.
```

矩阵分解：

```math
\hat r_{ui}=u_u^\top v_i.
```

双塔对比损失：

```math
\mathcal L
=-\log
\frac{e^{s(q,i^+)/\tau}}
{\sum_{j\in\{i^+\}\cup\mathcal N_q}e^{s(q,j)/\tau}}.
```

BM25：

```math
\sum_{t\in q}\operatorname{IDF}(t)
\frac{tf(t,d)(k_1+1)}
{tf(t,d)+k_1(1-b+b|d|/\operatorname{avgdl})}.
```

### 排序

LogLoss：

```math
-y\log p-(1-y)\log(1-p).
```

Pairwise logistic loss：

```math
-\log\sigma(s^+-s^-).
```

DCG：

```math
\operatorname{DCG@K}
=\sum_{i=1}^{K}
\frac{2^{rel_i}-1}{\log_2(i+1)}.
```

MMoE：

```math
h_t(x)=\sum_e g_{t,e}(x)f_e(x).
```

FM：

```math
\hat y
=w_0+\sum_iw_ix_i
+\sum_{i<j}\langle v_i,v_j\rangle x_ix_j.
```

DIN：

```math
u(q)=\sum_j\alpha(h_j,q)h_j.
```

### 列表与生成

MMR：

```math
\arg\max_i
\left[\theta r_i-(1-\theta)
\max_{j\in S}\operatorname{sim}(i,j)\right].
```

DPP：

```math
P(S)\propto\det(L_S).
```

生成式检索：

```math
P(d\mid q)=\prod_tP(d_t\mid d_{<t},q).
```

列表生成：

```math
P(i_1,\ldots,i_m\mid H)
=\prod_tP(i_t\mid i_{<t},H).
```

## 两周复习安排

### 第一周：传统链路

第 1 天：第 1-2 章。能照总图讲完一次请求，并说明日志怎样变成样本和特征。

第 2 天：第 3 章。手写 BM25、ItemCF、矩阵分解，讲清离线索引。

第 3 天：第 4-5 章。重点是负样本、ANN、多路召回和 query 改写。

第 4 天：第 6 章。对比 pointwise、pairwise、listwise，掌握 NDCG。

第 5 天：第 7-8 章。多目标、校准、FM/DCN 和粗排。

第 6 天：第 9-11 章。DIN/SIM、MMR/DPP、冷启动与探索。

第 7 天：闭卷画一次短视频推荐和电商搜索架构，再说明怎样做离线验收和 A/B。

### 第二周：生成式与系统题

第 8 天：第 12 章。对比 DSI、NCI、SEAL、TIGER，手写 Semantic ID 量化过程。

第 9 天：第 13 章前半。对比 pointwise、pairwise、listwise LLM reranking。

第 10 天：HSTU 与 OneRec。回答生成式推荐是否会替代级联。

第 11 天：第 14 章。画标准 RAG、Self-RAG 和 search agent。

第 12 天：做生成式搜索案例，重点回答引用、评价、安全和成本。

第 13 天：闭卷串讲生成式检索、生成式推荐和 agentic search，补齐不熟悉的公式。

第 14 天：完整模拟。先问业务和约束，再设计系统；最后才选择模型。
