# 查询词推荐

## 第 16 章 查询词推荐

搜索排序决定"用户搜了以后看到什么"，查询词推荐决定"用户会搜什么"。后者有自己的召回、排序和评价闭环，不能当成搜索框旁边的一条小规则。

### 16.1 四个场景

| 场景 | 当前可用信号 | 主要目的 |
| --- | --- | --- |
| 搜前推词 | 用户历史、热点、场景 | 激发一次原本不会发生的搜索 |
| SUG | 已输入前缀、用户与上下文 | 减少输入成本，补全或纠错 |
| 结果页推词 | 当前 query、用户 | 继续深挖或拓宽需求 |
| 文档内推词 | 当前文档、用户 | 从消费内容转入相关搜索 |

搜前推词和文档内推词能提高搜索渗透率。SUG 发生时用户已经决定搜索，主要价值是效率。结果页推词发生在一次搜索之后，主要增加搜索深度和兴趣宽度。

### 16.2 词库

候选词通常来自：

- 人工审核的精品词库；
- 站内高频 query；
- 优质文档标题和 OCR 文本；
- 外部热点与实体库；
- 生成模型产出的候选。

入库前要检查安全、低质、时效和供给。一个 query 很吸引点击，但搜索结果页没有足够相关文档，推给用户只会把问题从推词页转移到结果页。

### 16.3 SUG 召回

SUG 延迟预算很紧，成熟做法仍以确定性索引为主：

- 前缀召回：`"上海迪" -> "上海迪士尼"`；
- 拼音前缀和拼音简写；
- 分词包含与首字匹配；
- 核心词召回；
- 同音字和输入法纠错；
- 低召回量时的生成式扩展。

激进通道应在基础通道不足时触发。生成 `"日语慢速教学 -> 日语慢速教学楼"` 字面合法但语义荒谬，需要用词共现、相关性和供给量过滤。

SUG 索引要支持高频更新和热点插入。纯离线日更会错过刚发生的事件。

### 16.4 Q2Q

Q2Q 用当前 query 召回相关 query。

协同方法把 query 当 item。若许多用户先后搜索 `q_1` 和 `q_2`，二者可能属于同一兴趣：

```math
\operatorname{sim}(q_1,q_2)
=\frac{|U(q_1)\cap U(q_2)|}
{\sqrt{|U(q_1)|\,|U(q_2)|}}.
```

Swing 会降低小圈子用户对相似度的贡献，避免群体刷词制造假关联。

双塔 Q2Q 使用 query 文本、类目和核心词编码。正样本可来自结果页推词点击，负样本混合全局随机、batch 内和 hard negatives。文本双塔擅长语义邻近，协同方法能发现 `"梅西"` 与 `"阿根廷国家队"` 这类字面不近但兴趣相关的词，两者适合并行。

### 16.5 D2Q

D2Q 在文档发布后建立：

```text
document_id -> related queries
```

可用三种信号：

- 生成模型根据文档生成 query，再由相关性模型过滤；
- 双塔从 query 词库反向做向量检索；
- 行为日志统计"阅读文档后短时间内搜索了什么"。

生成和 Cross-BERT 过滤适合近线队列，不必阻塞发布请求。索引要带文档和模型版本，文档修改或过期后重新计算。

行为 D2Q 必须限制时间窗口和后续 query 数量。用户两天后搜的任意内容不一定由当前文档激发。

### 16.6 各场景怎样拼召回

搜前推词可使用：

```text
U2Q2Q：用户历史 query -> Q2Q -> 候选 query
U2D2Q：用户历史文档 -> D2Q -> 候选 query
热门与运营池
```

种子不能只取最近 N 条。保留一部分近期种子，再从长期历史按类目平衡抽样，能兼顾即时兴趣和兴趣宽度。

结果页推词以当前 query 做 Q2Q。文档内推词先做 D2Q，也可追加一跳 D2Q2Q 扩展，但跳数越多越容易语义漂移。

### 16.7 排序：点击只是第一段转化

推词排序至少预估两段：

```text
候选 query 被点击
  -> 搜索结果页是否有点击 / 交互
```

一个诱人的生僻 query 可能推词 CTR 很高，结果页却没有供给。排序分数可组合：

```math
s(q)
=w_1 P(\text{query click})
+w_2 P(\text{SERP success}\mid \text{query click})
+w_3 \operatorname{quality}(q)
+w_4 \operatorname{freshness}(q).
```

SUG 还要重视点击位置和输入长度，因为它是效率工具。搜前和结果页推词更像推荐，需要个性化、多样性、已搜过滤和频控。

### 16.8 多样性

连续推荐同义 query 会增加点击概率，却没有拓宽信息需求。可在类目、实体和 embedding 上做 MMR 或规则打散：

- 同实体不同属性可保留；
- 纯同义词只保留一个；
- 高风险和强时效 query 控制配额；
- 已搜索 query 在短窗口内降权；
- 搜前推词避免长期被单一兴趣占满。

多样性不能让不相关 query 获得加分。先过相关性与供给阈值，再在合格候选中做打散。

### 16.9 评价指标

| 指标 | 适用场景 | 含义 |
| --- | --- | --- |
| 推词有点比 | 全部 | 一次展示是否至少点击一个词 |
| 点击位置 | SUG | 用户找到目标词需要扫多少位置 |
| 输入长度 | SUG | 节省了多少输入 |
| SERP 有点比/首点位置 | 全部 | 下游搜索是否承接住 |
| 搜索渗透率 | 搜前、文档内 | 是否激发更多用户使用搜索 |
| 搜索深度 | 结果页推词 | session 内不重复 query 数 |
| 兴趣宽度 | 搜前、结果页 | query 类目是否扩展 |

推词实验必须同时看下游 SERP 转化。只优化推词 CTR，很容易学成标题党 query 推荐器。

### 16.10 本章自测

1. 为什么 SUG 和搜前推词不能用完全相同的评价指标？
2. Q2Q 的协同方法比纯文本相似多发现什么关系？
3. 行为 D2Q 为什么需要时间窗口？
4. 推词排序为什么要预估 SERP success？
5. D2Q2Q 为什么不能无限增加跳数？
6. 推词多样性应在哪个条件之后做？

<details>
<summary>参考答案</summary>

1. SUG 是输入工具，点击位置和输入长度直接反映效率；搜前推词负责激发搜索，更关心有点比和搜索渗透率。
2. 它能发现用户兴趣上的共现关系，即使两个 query 字面和向量并不相近。
3. 时间隔得太久或取太多后续 query，会把用户自然发生的其他需求误认成由文档激发。
4. Query 被点击后仍可能没有相关文档。加入下游成功率可避免推荐只有吸引力、没有内容供给的词。
5. 每一跳都会扩大候选和噪声，协同或语义关系会逐渐偏离当前文档。
6. 先满足安全、相关性、质量和供给阈值，再对合格候选打散；多样性不应救活不合格 query。
