## Chapter 11: Feature Interaction, Coarse Ranking, and Personalization

### 11.1 Business Patterns Often Lie in Interaction Terms

Looking at "user age" or "content category" alone is insufficient; the model needs to capture "the preference of a certain age group for a certain type of content at a certain time." The number of sparse feature combinations is enormous, and manual enumeration quickly spirals out of control.

### 11.2 FM

Factorization Machine uses low-rank vectors for second-order interactions:

```math
\hat y
=w_0+\sum_i w_ix_i
+\sum_{i<j}\langle v_i,v_j\rangle x_ix_j.
```

If `x` is one-hot, most dimensions are zero, and calculation only involves non-zero features. Inner products share statistical strength; even if a pair of features rarely appears together, reasonable interactions can be learned through their respective embeddings.

FM expresses second-order interactions. More complex relationships require deep networks or explicit high-order structures.

### Quick Coding: FM Forward Pass

Implement an FM forward pass without using nested feature loops. Use the sum-of-squares identity to reduce second-order interaction from `O(d²k)` to `O(dk)`, and verify the results using naive pairwise calculation.

Implementation:

```python
def fm_predict(x, bias, linear_weights, factors):
    ...
```

Where `factors[i]` is the `latent_dim`-dimensional vector for the `i`-th feature. You are required to use the following equivalent formula to reduce the second-order interaction complexity from `O(d²k)` to `O(dk)`:

```math
\frac{1}{2}\sum_f
\left[
\left(\sum_i v_{i,f}x_i\right)^2
-\sum_i(v_{i,f}x_i)^2
\right].
```

Raise a `ValueError` if input dimensions are inconsistent.

<details>
<summary>Reference answer</summary>

```python
def fm_predict(x, bias, linear_weights, factors):
    if len(x) != len(linear_weights) or len(x) != len(factors):
        raise ValueError("feature dimensions do not match")
    if not factors:
        return float(bias)

    latent_dim = len(factors[0])
    if any(len(vector) != latent_dim for vector in factors):
        raise ValueError("factor dimensions do not match")

    linear = sum(weight * value for weight, value in zip(linear_weights, x))
    interaction = 0.0

    for latent in range(latent_dim):
        summed = sum(factors[i][latent] * x[i] for i in range(len(x)))
        squared = sum(
            (factors[i][latent] * x[i]) ** 2
            for i in range(len(x))
        )
        interaction += 0.5 * (summed ** 2 - squared)

    return bias + linear + interaction
```

For `d` features and latent dimension `m`, time is `O(dm)` and auxiliary space is `O(1)`.

</details>

### 11.3 DCN

The cross layer of DCN is often written as:

```math
x_{l+1}
=x_0(x_l^\top w_l)+b_l+x_l.
```

Each layer retains the original input `x_0`, constructing higher-order explicit interactions layer by layer. Parallel deep networks learn implicit relationships, which are then concatenated.

DCN adds structured multiplicative interactions to the model, with fewer parameters than brute-force high-order combinations. Remembering it as "one layer more than an MLP" misses this point.

### 11.4 LHUC, SENet, and FiBiNET

LHUC performs conditional scaling on hidden units:

```math
h' = a(u, c)\odot h,
```

where `a` is generated by the user or scenario. It allows the same base to use different capacities for different users, domains, or scenarios.

LHUC began in speech recognition. Kuaishou applied the same user-conditioned hidden-unit scaling idea to recommendation ranking under the name PPNet. The useful questions are which personalization features produce the gate, which layer it scales, and whether the gate and backbone are trained together.

SENet first generates weights based on the entire set of features to recalibrate each field. FiBiNET then performs bilinear interaction on field embeddings. They are suitable for CTR scenarios with many fields where importance varies by sample.

Do not treat these models as a checklist of abbreviations to memorize. Remembering three questions is more practical:

1. What interaction does it explicitly model?
2. How are parameters shared across features or populations?
3. How much online computation and feature dependency is added?

### 11.5 Coarse Ranking

Coarse ranking sits between retrieval and re-ranking. Candidates are still numerous, so the model must be inexpensive; however, using only retrieval scores leads to false negatives.

Common strategies:

- Small MLP/three-tower models;
- Re-ranking distillation;
- Using only batch-readable features;
- Feature selection and low-precision inference;
- Layered top-k or early exit.

A three-tower pre-ranker separates:

```text
user tower: profile + context, evaluated once
item tower: static item features, cached when possible
cross tower: live statistics and interactions, small but evaluated per candidate
```

It does not remove all per-candidate work. The cross tower and upper multi-task network still run `N` times. Savings come from amortizing the large user tower and caching relatively static item outputs. Item-profile updates must invalidate caches, while dynamic statistics should not be cached as if they were static.

When evaluating coarse ranking, AUC alone is insufficient. The better question is: "Given the computational budget, how many candidates that would have been selected by re-ranking were preserved?" This can be answered using top-k consistency, re-ranking top-N recall, and online false-negative analysis.

### 11.6 How Personalized Features Enter the Model

`user_id` embeddings are strong for active users but powerless for new users and cross-domain users. Robust personalization typically combines:

- ID and statistical features;
- Long-term and short-term behavior;
- Population and scenario;
- Content semantics;
- Real-time intent.

Ranking features can be audited by source:

- user profile: ID, activity, demographics, long-term category interests;
- item profile: ID, author, category, text, multimodal quality, publication time;
- user statistics: impressions and interactions by time window, format, and category;
- item/author statistics: exposure and conversion by cohort and window;
- context: time, geography, device, and entry point;
- interactions: historical preference for the current category or author.

Counts often use `log(1+x)`, rates need smoothing, and missing features need coverage reports. A feature available for only 20% of users has a diluted aggregate effect; inconsistent training and serving defaults create direct distribution shift.

The stronger the personalization, the more one must be mindful of privacy, filter bubbles, and feedback loops. In search, explicit queries must be placed before user history; long-term profiles should not override current needs.

1. Each feature has a low-dimensional vector, and pairwise inner products share statistical strength across sparse combinations.
2. A cross layer explicitly builds bounded-order feature products while retaining the input. An MLP learns interactions implicitly.
3. It is useful for lightweight conditioning by user, scenario, or domain, such as multi-scenario ranking; it does not replace missing core features.
4. Coarse ranking must preserve items that fine ranking would select. Standalone AUC is insufficient if downstream top items are removed.
5. A query states current explicit intent. Personalization may reorder relevant candidates but should not promote irrelevant results.
