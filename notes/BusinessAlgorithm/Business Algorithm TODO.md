### 1.1 从一次请求开始

线上链路从一个带唯一 `request_id` 的请求开始。推荐请求至少包含用户、场景、时间、设备和近期行为；搜索还会带 query、过滤条件和地域。一个简化输入是：

```text
request_id
user_id / session_id
query 或当前入口
最近行为序列
时间、地域、设备、实验桶
```

以短视频首页为例，系统可能在 150 ms 内从亿级视频中返回 20 条：

```text
亿级内容
  -> ItemCF / 双塔 / 关注 / 热门等通道召回 3,000 条
  -> 去重、已看过滤、安全与地域过滤，剩 1,800 条
  -> 粗排保留 300 条
  -> 精排计算点击、时长、负反馈等分数，保留 80 条
  -> 重排控制作者重复、探索和广告位置
  -> 返回 20 条并记录完整曝光日志
```

这些数字会随产品和延迟预算变化，拆层逻辑不变：越靠前候选越多，单个候选能用的计算越少；越靠后候选越少，模型交互和列表约束越细。

搜索使用相似漏斗，但 query 是强输入。召回要同时处理精确词、语义和属性条件，排序必须守住 query-item 相关性。推荐主要从历史与场景推断意图，可以留出少量探索位置。

### 1.2 传统架构为什么采用多阶段

传统系统把召回、粗排、精排和重排拆开，最直接的原因是算力不够均摊。倒排表、相似表和 ANN 可以在亿级库上快速取回几千条；实时特征和复杂交叉只能留给后面的几百条候选。

各层的错误代价也不同。召回漏掉一个 item，后面的模型没有补救机会，所以这一层偏向覆盖；精排面对的是已经入围的候选，目标才是把细小偏好分开。

库存、地域、安全、广告频控和作者去重通常确定执行。它们留在模型外，原因很实际：规则需要审计、热修复和独立回滚。

级联架构也有代价。不同阶段各自训练，目标可能冲突；前级误杀无法恢复；多套特征、模型和 RPC 还会增加维护与通信成本。端到端生成式系统正是从这些问题切入，但几项代表工作的"端到端"范围并不相同。

- [HSTU / Generative Recommenders](https://proceedings.mlr.press/v235/zhai24a.html) 把推荐改写为序列转导，用一个可扩展模型根据用户动作序列预测后续动作或 item。它统一了推荐建模方式和训练扩展路径。论文没有据此声称库存、安全规则或所有线上候选服务都已消失。
- [OneRec](https://arxiv.org/abs/2502.18965) 明确用单个 encoder-decoder 替代 retrieve-and-rank 级联，从用户历史直接生成 session 级 item 列表，并用 DPO 做偏好对齐。它统一了召回、前排和排序目标，但仍需要 Semantic ID、reward model、约束解码和线上规则配合。
- [OneSearch](https://openreview.net/forum?id=JKGgHY9FKa) 面向电商搜索，让 query、用户行为和 item Semantic ID 进入统一生成框架，替代传统的 recall、pre-ranking、ranking 漏斗。搜索仍要处理 query-item 强相关性、库存和商品属性；论文系统中也保留了 item 编码与额外的 reward model selection。

讨论"端到端"时要说清边界：模型统一了哪些训练目标和在线阶段，哪些索引、规则与容灾路径仍在外部。否则同一个词会指向完全不同的系统。

### 1.3 按请求顺序拆系统

后续章节按图 1-1 的在线顺序展开：

| 阶段 | 输入 | 输出 | 后面讲什么 |
| --- | --- | --- | --- |
| 数据与请求 | 用户、query、item、上下文、历史日志 | 可复现的样本与特征 | 第 2 章 |
| 召回 | 全库、倒排索引、相似表、向量索引 | 数千候选及通道归因 | 第 3-5 章 |

| 列表决策 | 排序结果、规则和探索预算 | 最终展示列表 | 第 10-11 章 |
| 生成式路径 | query/历史与 item/doc 标识 | 生成的 item、SID 或答案 | 第 12-14 章 |
| 系统验证 | 全链路日志、版本和实验流量 | 上线结论与排障定位 | 第 15 章 |

读论文或设计方案时，沿着表格问四件事：输入是什么，替换哪一层，输出交给谁，失败后怎么降级。网络结构只是其中一项。

### 1.4 离线与在线边界

图 1-1 下半部分是离线与准实时链路。ItemCF 相似表、物品向量、Semantic ID 和倒排索引提前构建；用户短期行为、当前 query、库存和设备状态在请求时读取。两边通过版本号对齐：

```text
model_version
feature_schema_version
index_version
rule_version
experiment_id
```

一次曝光若没有这些字段，指标变化后很难判断是模型、索引、特征还是规则更新造成的。

一张能用于排障的架构图还要标出：

1. 离线日志和样本生成；
2. 模型训练与索引构建；
3. 在线特征服务；
4. 召回、排序和重排；
5. 曝光与交互日志回流。

曝光与交互日志若没有回流，训练样本、线上指标和版本发布无法对齐。

## 第 2 章 数据、样本与特征流

训练表来自三类原始记录：线上请求、每层候选轨迹和延迟到达的用户反馈。先把它们按时间与版本对齐，后面才谈得上召回 pair、曝光样本或列表偏好。

### 2.1 请求是所有数据的主键

同一次请求会经过多个服务。`request_id` 要贯穿 query 理解、召回、排序、重排和曝光，才能还原一件物品在哪一层进入、在哪一层被删掉。

推荐请求常见字段：

```text
request_id, user_id, session_id
scene, timestamp, device, region
recent_actions
experiment_ids
```

搜索还需要：

```text
raw_query, normalized_query
query_intent, filters
rewrite_source
```

用户画像和历史序列不宜整份复制进每条日志。日志记录快照版本或 feature key，需要时再按 point-in-time join 还原。

### 2.2 候选轨迹要能逐层回放

只记最终曝光不够。召回调优需要知道每个候选来自哪一路，粗排需要知道哪些候选被提前误杀，规则排障需要保存过滤原因。

一条候选轨迹至少包含：

| 字段 | 用途 |
| --- | --- |
| `request_id, item_id` | 连接一次请求中的同一候选 |
| `stage, channel` | 标记召回、粗排、精排、重排及召回来源 |
| `raw_score, calibrated_score` | 回放模型分数和融合过程 |
| `rank_before, rank_after` | 观察每一层怎样改变位置 |
| `filter_reason` | 区分库存、安全、已看、配额等删除原因 |
| `model/index/rule_version` | 定位发布与版本错配 |
| `event_time` | 保证时间切分和特征回放 |

全量保存每层所有候选可能太贵，可以对中间轨迹采样；最终曝光、关键过滤和实验流量应保留完整记录。

### 2.3 反馈要按时间成熟

曝光后很快能观察点击，购买、退款和次日留存要等更久：

```text
t0 请求与曝光
t1 点击 / 跳过
t2 加购 / 播放完成
t3 下单
t4 退款或留存窗口结束
```

训练样本要声明标签截止时间。例如预测 7 日购买，今天的曝光不能明天就当负例。常见做法是只使用已经过成熟窗口的样本，或对尚未成熟的标签单独建模。

行为强度也不能混成一个 `label`。点击、有效播放、点赞、购买和举报有不同含义；后面的多目标模型会分别消费这些字段。

### 2.4 未点击不自动等于负样本

用户只能反馈系统展示过的内容。未召回 item 没有观察标签，曝光未点击也可能因为位置太低、页面没滚到或网络中断。

需要保留几类偏差：

- 选择偏差：旧召回与排序策略决定了谁能进入日志；
- 位置偏差：头部候选天然获得更多交互；
- 热门偏差：高曝光继续积累更多正反馈；
- 生存者偏差：被规则提前过滤的内容从训练集消失；
- 延迟反馈：标签尚未成熟就被误记为负例。

随机曝光、探索桶、propensity weighting 和延迟修正可以缓解部分问题。负采样只改变训练分布，无法凭空补出从未展示候选的真实反馈。

### 2.5 各阶段需要不同样本

| 模块 | 一条训练样本 | 常见正例 | 候选或负例 |
| --- | --- | --- | --- |
| 协同召回 | 用户历史与下一个 item | 点击、有效播放、购买 | 全库采样、批内或 hard negative |
| 搜索召回 | query 与 document/item | 人工相关或高质量交互 | BM25/旧模型召回的难例 |
| 精排 | 一次曝光中的 user/query-item | 点击、时长、转化 | 同次曝光未互动候选 |
| 列表模型 | 请求与完整 slate | 列表级反馈 | 其他排列或策略生成列表 |
| 生成式推荐 | 历史序列与 item/SID 序列 | 下一 item 或偏好列表 | 词表竞争者、rejected、rollout |

同一条用户行为可以进入多个任务，但样本单位不同。把召回 pair、精排曝光样本和列表偏好对混在一张表里，通常会丢掉候选来源与列表上下文。

### 2.6 特征按可用时间拼接

特征大致分为四组：

- 用户：长期画像、近期行为、活跃度；
- query 或场景：词项、意图、时间、地域、入口；
- item/doc：内容、类目、作者、质量、库存；
- 交叉：用户对类目的历史偏好、query-item 文本匹配、上下文中的实时统计。

训练时只能读取事件发生前已经存在的值。若用当天最终销量预测当天早上的点击，就发生了时间穿越。可靠的特征平台要支持 point-in-time join，并记录特征生成时间、schema 和默认值。

线上缺失处理也要与训练一致。训练中删除缺失样本，服务时却填零，会让模型上线后看到新的输入分布。

### 2.7 数据集按时间切分

推荐与搜索日志有明显时间顺序。随机打散会让未来热门 item、后续用户行为或新版特征泄漏进训练集。

```text
训练窗口        验证窗口       测试窗口
──────────────|───────────|───────────>
              冻结特征和样本构造版本
```

每次实验应能回答：用了哪段日志、标签何时成熟、负样本怎样采、特征快照是哪版、候选由哪个策略产生。没有这些信息，离线结果很难复现。
