# 数据、样本与特征流

## 第 2 章 数据、样本与特征流

训练表来自三类原始记录：线上请求、每层候选轨迹和延迟到达的用户反馈。先把它们按时间与版本对齐，后面才谈得上召回 pair、曝光样本或列表偏好。

### 2.1 请求是所有数据的主键

同一次请求会经过多个服务。`request_id` 要贯穿 query 理解、召回、排序、重排和曝光，才能还原一件物品在哪一层进入、在哪一层被删掉。

推荐请求常见字段：

```text
request_id, user_id, session_id
scene, timestamp, device, region
recent_actions
experiment_ids
```

搜索还需要：

```text
raw_query, normalized_query
query_intent, filters
rewrite_source
```

用户画像和历史序列不宜整份复制进每条日志。日志记录快照版本或 feature key，需要时再按 point-in-time join 还原。

### 2.2 候选轨迹要能逐层回放

只记最终曝光不够。召回调优需要知道每个候选来自哪一路，粗排需要知道哪些候选被提前误杀，规则排障需要保存过滤原因。

一条候选轨迹至少包含：

| 字段 | 用途 |
| --- | --- |
| `request_id, item_id` | 连接一次请求中的同一候选 |
| `stage, channel` | 标记召回、粗排、精排、重排及召回来源 |
| `raw_score, calibrated_score` | 回放模型分数和融合过程 |
| `rank_before, rank_after` | 观察每一层怎样改变位置 |
| `filter_reason` | 区分库存、安全、已看、配额等删除原因 |
| `model/index/rule_version` | 定位发布与版本错配 |
| `event_time` | 保证时间切分和特征回放 |

全量保存每层所有候选可能太贵，可以对中间轨迹采样；最终曝光、关键过滤和实验流量应保留完整记录。

### 2.3 反馈要按时间成熟

曝光后很快能观察点击，购买、退款和次日留存要等更久：

```text
t0 请求与曝光
t1 点击 / 跳过
t2 加购 / 播放完成
t3 下单
t4 退款或留存窗口结束
```

训练样本要声明标签截止时间。例如预测 7 日购买，今天的曝光不能明天就当负例。常见做法是只使用已经过成熟窗口的样本，或对尚未成熟的标签单独建模。

行为强度也不能混成一个 `label`。点击、有效播放、点赞、购买和举报有不同含义；后面的多目标模型会分别消费这些字段。

### 2.4 未点击不自动等于负样本

用户只能反馈系统展示过的内容。未召回 item 没有观察标签，曝光未点击也可能因为位置太低、页面没滚到或网络中断。

需要保留几类偏差：

- 选择偏差：旧召回与排序策略决定了谁能进入日志；
- 位置偏差：头部候选天然获得更多交互；
- 热门偏差：高曝光继续积累更多正反馈；
- 生存者偏差：被规则提前过滤的内容从训练集消失；
- 延迟反馈：标签尚未成熟就被误记为负例。

随机曝光、探索桶、propensity weighting 和延迟修正可以缓解部分问题。负采样只改变训练分布，无法凭空补出从未展示候选的真实反馈。

### 2.5 各阶段需要不同样本

| 模块 | 一条训练样本 | 常见正例 | 候选或负例 |
| --- | --- | --- | --- |
| 协同召回 | 用户历史与下一个 item | 点击、有效播放、购买 | 全库采样、批内或 hard negative |
| 搜索召回 | query 与 document/item | 人工相关或高质量交互 | BM25/旧模型召回的难例 |
| 精排 | 一次曝光中的 user/query-item | 点击、时长、转化 | 同次曝光未互动候选 |
| 列表模型 | 请求与完整 slate | 列表级反馈 | 其他排列或策略生成列表 |
| 生成式推荐 | 历史序列与 item/SID 序列 | 下一 item 或偏好列表 | 词表竞争者、rejected、rollout |

同一条用户行为可以进入多个任务，但样本单位不同。把召回 pair、精排曝光样本和列表偏好对混在一张表里，通常会丢掉候选来源与列表上下文。

### 2.6 特征按可用时间拼接

特征大致分为四组：

- 用户：长期画像、近期行为、活跃度；
- query 或场景：词项、意图、时间、地域、入口；
- item/doc：内容、类目、作者、质量、库存；
- 交叉：用户对类目的历史偏好、query-item 文本匹配、上下文中的实时统计。

训练时只能读取事件发生前已经存在的值。若用当天最终销量预测当天早上的点击，就发生了时间穿越。可靠的特征平台要支持 point-in-time join，并记录特征生成时间、schema 和默认值。

线上缺失处理也要与训练一致。训练中删除缺失样本，服务时却填零，会让模型上线后看到新的输入分布。

### 2.7 数据集按时间切分

推荐与搜索日志有明显时间顺序。随机打散会让未来热门 item、后续用户行为或新版特征泄漏进训练集。

```text
训练窗口        验证窗口       测试窗口
──────────────|───────────|───────────>
              冻结特征和样本构造版本
```

每次实验应能回答：用了哪段日志、标签何时成熟、负样本怎样采、特征快照是哪版、候选由哪个策略产生。没有这些信息，离线结果很难复现。

### 2.8 本章自测

1. 为什么 `request_id` 必须贯穿整条在线链路？
2. 只记录最终曝光会漏掉哪些召回和粗排问题？
3. 7 日转化标签为什么需要成熟窗口？
4. 曝光未点击在什么条件下才适合作为负例？
5. point-in-time join 解决什么问题？

<details>
<summary>参考答案</summary>

1. 它把请求、各阶段候选、最终曝光和延迟反馈连接成一次可回放事件；缺失后无法判断候选在哪一层进入或被过滤。
2. 会丢掉召回通道、粗排误杀、过滤原因和阶段排名变化，只能看到幸存候选，无法定位漏斗损失。
3. 窗口结束前尚未购买的样本仍可能在第 2–7 天转化。过早标负会制造系统性假负例。
4. 至少要确认用户实际看到了该位置，并控制位置、页面深度和网络中断等偏差。未曝光 item 不能直接当作已观察负例。
5. 它保证训练样本只读取事件发生时已经存在的特征，防止未来销量、后续行为或新版画像泄漏进历史样本。
