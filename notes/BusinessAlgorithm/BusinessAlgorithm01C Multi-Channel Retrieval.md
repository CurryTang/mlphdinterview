# Query、内容和多路召回

## 第 6 章 Query、内容和多路召回

### 6.1 Query 处理决定召回边界

搜索 query 短、歧义多，输入质量直接决定召回上限。

分词把连续字符串拆成可检索 term。词权重判断"苹果 手机 壁纸"里哪个词决定意图。类目识别决定去商品、视频还是百科索引。意图识别还会控制时效、地域、成人安全等下游模块。

这些模块不一定都是独立模型。大型系统会共享 encoder，也会保留字典和规则处理型号、拼写、敏感词与强业务意图。

### 6.2 Query 改写

改写的目标很实在：让原本找不到的相关内容进入候选集。

常见类型：

- 拼写纠错；
- 同义词和别名扩展；
- 分词修正；
- 实体规范化；
- 省略补全；
- 意图扩展；
- 历史 query 关联。

原 query 与改写 query 通常分别召回，再合并候选。改写过强会引入语义漂移，所以需要原 query 保底、改写置信度和相关性约束。

在生成式搜索中，query rewrite 又多了一层：复杂问题会被拆成多个子问题，搜索 agent 根据中间证据继续改写。第 19 章会回到这里。

### 6.3 推荐侧的内容召回

推荐没有 query，但仍可以从内容生成候选：

- 类目、标签和作者；
- 文本、图像、音频 embedding；
- 地域、语言和发布时间；
- 用户刚刚交互内容的相似项；
- 运营配置与热点池。

内容召回对新品和长尾尤其重要。协同信号尚未形成时，内容是少数可用信息。不过，"看起来相似"不等于"同一用户会喜欢"，线上通常让内容召回与协同召回并行。

课程里几条常见补充通道可以直接写成索引路径：

```text
GeoHash / 城市 -> 按时间倒排的优质内容
user -> 关注或近期交互的 author -> author 最新内容
user -> 感兴趣的 author -> 相似 author -> 最新内容
```

第一类几乎不个性化，负责本地与同城供给；后两类是 U2A2I，显式关注精度高，隐式交互和相似作者负责扩大覆盖。

### 6.4 多路召回怎么合并

假设有 ItemCF、双塔、热门、关注作者、地理位置五路候选。直接拼接会遇到：

- 不同通道分数量纲不同；
- 同一物品重复出现；
- 强通道吃光配额；
- 小通道的增量候选被淹没；
- 过滤后某些通道几乎为空。

线上通常组合固定或动态配额、通道内截断、分数归一化、学习融合和去重回填。融合日志必须保留候选的全部召回来源，否则只能看到最后采用的通道，算不出单路覆盖、重叠率和增量转化。

一个通道独立 Recall 不高，也可能有价值。如果它召回的正例是其他通道都找不到的长尾内容，增量价值很大。

### Quick Coding：RRF 融合

不用校准不同召回通道的原始分数，只根据各通道名次实现 Reciprocal Rank Fusion。记得处理单个列表中的重复候选和稳定 tie-break。

实现：

```python
def reciprocal_rank_fusion(rankings, rrf_k=60, top_n=None):
    ...
```

每个 ranking 是一个候选 ID 列表。融合分数：

```math
score(d)=\sum_m\frac{1}{rrf\_k+rank_m(d)}.
```

rank 从 1 开始。同一 ranking 中重复候选只计算第一次出现。按分数降序、候选 ID 升序返回。

<details>
<summary>参考答案</summary>

```python
from collections import defaultdict


def reciprocal_rank_fusion(rankings, rrf_k=60, top_n=None):
    scores = defaultdict(float)

    for ranking in rankings:
        seen = set()
        for rank, candidate in enumerate(ranking, start=1):
            if candidate in seen:
                continue
            seen.add(candidate)
            scores[candidate] += 1.0 / (rrf_k + rank)

    result = sorted(scores.items(), key=lambda pair: (-pair[1], pair[0]))
    return result if top_n is None else result[:top_n]
```

若所有 ranking 的总长度为 `N`，计分是 `O(N)`，最终排序是 `O(M log M)`，`M` 为去重候选数。

</details>

### 6.5 曝光过滤与缓存召回

推荐通常要过滤用户近期看过或明确不喜欢的内容。过滤窗口太短会重复，太长会把可复看的优质内容永远排除。曝光过滤最好保留行为类型、时间和场景，而不是只维护一个永久黑名单。

用户已曝光集合很大时，常用 Bloom filter 做近似 membership test。它回答 `"一定没看过"` 或 `"可能看过"`：后者可能是假阳性，会误伤未曝光物品。若集合含 `n` 个元素、bit array 长度为 `m`、使用 `k` 个哈希函数，假阳性率近似为：

```math
\left(1-e^{-kn/m}\right)^k.
```

Bloom filter 原生不支持删除。按月只保留曝光窗口时，可使用按天/周滚动的多个 filter，过期后整体丢弃；若需要逐项删除，则使用 counting Bloom filter 或精确存储。`m` 和 `k` 要从可接受误伤率反推，不能只追求省内存。

推荐还有一种候选缓存：精排靠前但本轮没被重排曝光的内容暂存到用户缓存，下次作为一路召回。内容一旦曝光、超过重试次数、TTL 到期或缓存满时按策略淘汰。它复用了已经付过精排成本的候选，但必须重新检查库存、安全与时效。

搜索的缓存/KV 召回则利用高频 query 的稳定结果。它延迟低，但要处理时效、个性化和缓存污染。热点新闻 query 不能长期复用旧结果，个性化 query 也不应直接共享所有用户的缓存。

### 6.6 离线召回与反向召回

高频 query 的结果相对稳定，可以离线跑一条更重的搜索链路：

```text
高频 query
  -> 扩展召回
  -> 大模型相关性与质量排序
  -> 人工/规则过滤
  -> query -> top documents 缓存
```

线上命中后直接读 posting，延迟很低。缓存必须带 query 规范化、地域、时效和版本条件；热点变化或文档下架时要主动失效。

搜索的日志缓存不是简单保存某一次结果页。对每条头部 query 聚合过去若干天的 top 文档，融合出现频率、点击/交互频率、相关性、内容质量与文档年龄，建立：

```text
q -> [(d, score), ...]
```

它可以是非个性化索引，也可以按少量人群建立 `(q, group) -> documents`。人群切得太细会稀释统计并放大内存。索引按天把新旧分数做衰减归并；新召回通道做实验时，还要从缓存建库日志中排除实验结果，否则实验组发现的文档会被缓存泄漏给对照组。

离线搜索链路会在低峰时段为头部 query 扩大召回和打分量，并把精排相关性分数一并写入 KV。线上命中后可少算一部分非个性化召回与相关性。突发/强时效、地域 query、新文档和个性化通道仍要在线处理，不能被离线结果覆盖。

反向召回从文档出发找 query。文档发布后，可以用生成模型产生 Anchor Queries，或用双塔在 query 词库中反向 ANN，再用 Cross-BERT 过滤：

```text
document -> candidate queries -> relevance filter
         -> query -> document inverted postings
```

它适合长尾文档：用户尚未搜过这篇新文档，日志里没有 query-document 共现，但内容模型仍能把它挂到合适 query 下。

D2Q 的训练 pair 可从头部 query 的历史结果中筛：文档排名高、点击率高，而且精排相关性达到高档。先用海量日志 pair 训练，再用人工相关性 pair 微调。离线生成时过滤过短或低质文档，为每篇文档采样多条 query，再用相关性模型验收。两个直接指标是生成 pair 的相关性通过率，以及每篇文档平均产生多少条不重复的高相关 query。

改写还能接到缓存上：

```text
tail query q -> head rewrite q' -> cached documents d
```

此时既要验证 `(q,q')` 相关，也要验证 `(q',d)` 的缓存质量。

离线召回不能完全取代在线召回。库存、时效、用户上下文和刚发生的 query 仍需在线处理。常见组合是：

```text
原 query 在线召回
+ rewrite 在线召回
+ 离线 query posting
+ 缓存与热点通道
```

### 6.7 召回效果怎样验收

召回阶段先看目标候选有没有进入 top-K：

```math
\operatorname{Recall@K}
=\frac{\lvert \text{top-K 中的相关项}\rvert}
{\lvert \text{测试窗口中的相关项}\rvert}.
```

推荐里的分母通常是测试窗口中发生过的点击、有效播放或购买，并非用户全部潜在兴趣。搜索可以使用人工相关性标注，但要按头部、长尾、实体和复杂 query 分桶。一个总 Recall 很容易被高频样本掩盖。

多路召回还要看：

- 每路覆盖率与独立召回量；
- 通道之间的候选重叠；
- 新品、长尾和冷 query 的覆盖；
- 合并及过滤后的保留率；
- 在固定 P99 延迟下能返回多少候选。

指标下降时，沿候选漏斗回放：

1. ground-truth 是否进入任一通道？
2. 进入了，但通道内排名是否太低？
3. 合并配额是否把它截掉？
4. 是否被曝光、库存、安全规则过滤？
5. 是否进入排序后被误杀？

先定位候选在哪一步消失，再决定是否重训双塔。过滤窗口和通道配额出错时，换模型不会解决问题。

8. 新召回实验为什么可能被搜索缓存污染？

<details>
<summary>参考答案</summary>

1. 为 rewrite 标记来源和置信度，限制改写类型，并同时保留原 query 通道。离线按意图切片检查，线上观察无结果率、改写撤销率和坏例。
2. 内容特征能给无行为 item 生成表示，但缺少协同偏好、流行度和人群反馈。单独使用容易得到语义相关却不符合用户兴趣的结果。
3. 在固定总候选预算下做关闭通道实验，观察它提供的独占正例、长尾覆盖和下游增量，而不是只看该通道自身 Recall。
4. BM25、余弦相似度、共现分数和规则分值的尺度与分布不同。应做校准、rank-based 融合或交给后级模型学习。
5. 头部 query 数量和曝光占比更大，平均指标会掩盖长尾退化。必须按频次、意图、语言等切片报告 Recall。
6. 文档发布后先生成或向量检索相关 query，再经相关性模型过滤，建立 `query -> document` posting，因此不依赖已有 query-document 点击日志。
7. 多个 item 的哈希位可能碰撞，使未曝光 item 的全部位恰好为 1。可按天/周维护多个 filter，过期后整体删除，或在需要逐项删除时使用 counting Bloom filter。
8. 实验组新召回的优质文档进入曝光日志后，缓存建库会把它收录并返回给对照组，造成组间泄漏。建库时应识别并排除实验候选，或让缓存按实验域隔离。

</details>
