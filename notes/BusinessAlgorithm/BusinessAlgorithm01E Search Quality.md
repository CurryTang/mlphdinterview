# 搜索体验与评价

## 第 7 章 搜索体验与评价

搜索结果好不好，不能只看文档点击率。用户可能因为标题党点击后立即退出，也可能不点击，因为答案已经在卡片里。评价搜索要先拆清楚用户满意度来自哪里，再选择能解释这些环节的指标。

### 7.1 影响满意度的五类信号

搜索排序常同时考虑：

1. 相关性：文档是否回答 query；
2. 内容质量：来源是否可信，内容是否有用；
3. 时效性：用户是否需要新内容；
4. 个性化：在相关结果内匹配用户偏好；
5. 地域性：结果是否符合用户当前或指定地点。

这五类信号不能合并成一个含义含糊的 `"quality_score"`。它们的标注、更新频率和失败方式不同，最好分别产出，再在排序阶段融合。

### 7.2 内容质量

课程中使用 EAT：专业性、权威性、可信度。Google 后来的公开表述常写作 E-E-A-T，多了 Experience。工程含义没有变：对医疗、金融、法律等 YMYL query，来源和作者资质比一般娱乐 query 更重要。

质量模型通常拆成多个分数：

- 作者或站点信誉；
- 事实性与专业性；
- 原创度和信息量；
- 标题党、关键词堆砌和软广风险；
- 图片或视频清晰度；
- 安全、低俗和欺诈风险。

这些分数大多可在文档发布时离线或近线计算，写入 document profile。排序请求直接读取，避免为每个 query 重跑昂贵模型。

### 7.3 时效性

时效性不是统一的 `"越新越好"`。

- 突发时效：事件刚发生，query 搜索量或相关内容发布量突然上升；
- 一般时效：从文字就能判断用户偏好近期信息，例如 `"现在办护照流程"`；
- 周期时效：高考、春运、双十一等每年在固定窗口变热；
- 无时效：稳定知识或历史内容，新旧通常不影响价值。

突发时效主要靠数据挖掘。只看 `"阿根廷"` 的文字，模型不知道此刻是否刚结束世界杯决赛。一般时效可以用语义模型判断。

下游需要共同承接：

```text
时效意图
  -> 新文档索引 / 热点召回
  -> 质量与相关性过滤
  -> age decay 或 freshness boost
  -> 过期检测与降权
```

只在精排加发布时间特征，救不回召回阶段漏掉的突发内容。

### 7.4 地域性

地域搜索要区分 query 中的地点和用户当前位置：

```text
"上海徐汇咖啡" -> 显式地点=上海徐汇
"附近咖啡"     -> 地点来自用户定位
"迪士尼住宿"   -> POI=迪士尼，存在隐式地域需求
```

典型链路包括 POI 识别、地名规范化、地域索引召回、距离计算和排序融合。距离不是越近越好；用户搜景点攻略时内容质量更重要，搜 `"附近加油站"` 时距离权重更高。

地域模型至少要给出两个量：地域需求强度，以及对距离的敏感度。`"附近美食"` 的地域需求接近硬约束，`"火锅店"` 可能只需部分本地结果；步行找餐厅和开车周边游的可接受距离也不同。两者会分别影响地域候选配额和距离衰减。

地域实验应按定位权限、城市层级、显式/隐式地域 query 分桶。大量用户没有定位权限，整体均值会掩盖真正生效的人群。

### 7.5 个性化

宽泛 query 如 `"头像"` 有大量相关文档，个性化能帮助调序。精准 query 如 `"权力的游戏龙妈头像"` 已经把需求说得很清楚，长期画像不应压过当前意图。

实用约束是：

> 个性化在相关候选内排序，不负责把不相关内容变成相关。

CTR 和交互率模型也有非个性化价值。高质量且相关的文档通常更容易被点击，这些行为模型可补语义模型的 bad case，但要控制位置偏差和标题党。

### 7.6 北极星、过程指标和人工评价

搜索评价通常分三层：

| 层级 | 指标 | 回答的问题 |
| --- | --- | --- |
| 北极星 | Search DAU、搜索渗透率、LT7/LT30 | 用户是否长期愿意使用搜索 |
| 过程 | 有点比、首屏有点比、首点位置、换词率、交互 | 一次搜索哪里变好了 |
| 人工 | GSB、分档相关性、DCG/NDCG | 没有稳定行为标签时，结果页质量如何 |

`Search DAU / DAU` 是搜索渗透率。单个算法实验很难在短窗口内显著改变留存，因此过程指标负责快速诊断，长期 Holdout 再检查累计收益。

留存要写清统计口径。`次 n 日内留存` 是今天使用搜索的用户中，未来 `n` 天内至少再用一次的比例；`第 n 日留存` 只看第 `n` 天当天。前者随 `n` 单调不减，后者通常随 `n` 下降，而且都要等窗口成熟。留存率单独上涨也可能是低活用户流失后的分母变化，因此必须和 Search DAU 及分群规模一起看。

### 7.7 文档点击率和有点比

文档 CTR 的分母是文档曝光：

```math
\operatorname{DocCTR}
=\frac{\#\text{document clicks}}
{\#\text{document impressions}}.
```

有点比是 query 级成功率：

```math
\operatorname{QueryCTR}
=\frac{\#\text{searches with at least one click}}
{\#\text{searches}}.
```

一个结果页塞入更多相似文档，可能提高总点击数，却不一定提高有点比。对工具型搜索，有点比和首屏有点比通常比单文档 CTR 更接近"用户有没有找到东西"。

平均首点位置要给无点击请求设置截断位置，否则无点击样本会被丢掉，指标只在成功请求上自我感觉良好。

点击还可定义为有效点击：进入文档后停留超过阈值，或产生点赞、收藏、关注等后续行为。它能排除一部分误触和标题党，但阈值需要按内容形态校准。

浏览深度记录结果页最后曝光到的位置。用户已经在前几条找到答案时通常不会继续翻页，所以它经常是越小越好；探索型 query 可能天然浏览更深，必须按意图切片，不能跨场景比较一个总均值。

### 7.8 主动换词和交互

用户搜完 `"机械键盘"` 后很快改成 `"机械键盘 女生"`，可能说明原结果个性化不足。主动换词率应排除系统推荐的被动改写，并限制时间窗口和 query 相似度。

点赞、收藏、关注、购买等交互比点击强，但更稀疏。线上实验里可把多种交互按与留存的关系组合成稳定指标，同时保留单项护栏，避免一种高频行为吞掉其他信号。

交互指标的分母也决定含义：`每次搜索的点赞数` 衡量一条 query 最终带来多少互动，`每次点击的点赞数` 更接近点击后内容质量。两者不能混报。

用户日均 query 数有双重解释。换词变少会让它下降，搜索习惯增强又会让它上升，所以这个指标单独变好或变坏都不能下结论，必须结合主动换词率、Search DAU 和搜索深度。

### 7.9 Side-by-Side 与 GSB

人工 Side-by-Side 固定 query、用户画像和场景，同时展示新旧策略的两个结果页。标注者给出：

- Good：新策略更好；
- Same：两者相当；
- Bad：新策略更差。

常用汇总是：

```math
\operatorname{GSB}
=\frac{G-B}{G+S+B}.
```

抽样要覆盖头部、中频、长尾、时效、地域和高风险类目。只随机抽日志会被头部 query 支配。

人工评价能发现相关性和质量 bad case，但不能替代 A/B 测试。标注页面没有真实用户选择，也测不到延迟、个性化反馈和长期行为。

它本身也有成本：样本量通常只有几百条 query，结论受标注规范和人员经验影响；强个性化结果还要求标注者理解用户画像。上线决策若依赖 SBS，需要盲测、重复标注和一致性监控，不能看到一次 GSB 正向就推全。

### 7.10 本章自测

1. 为什么内容质量、相关性和时效性应分别建模？
2. 突发时效为什么不能只靠 BERT 判断？
3. 有点比和文档 CTR 的分母有什么不同？
4. 平均首点位置怎样处理无点击请求？
5. 个性化搜索为什么必须让 query 保持主导？
6. GSB 能替代线上 A/B 测试吗？
7. `次 7 日内留存` 和 `第 7 日留存` 有什么区别？
8. 浏览深度为什么通常越小越好，又为什么必须按意图切片？

2. 突发性取决于当前世界状态。只看 query 文本无法知道最近是否发生新闻，需要搜索量、发布量和外部热点等实时信号。
3. 文档 CTR 以文档曝光为分母；有点比以搜索请求为分母，只关心一次搜索是否至少产生一个点击。
4. 给无点击和超过阈值的位置统一记为截断值，使失败请求仍进入平均数。
5. Query 表达当前明确需求。画像只能在相关结果中调序，否则历史偏好会把不相关文档推上来。
6. 不能。GSB 是受控的人工体验比较，测不到真实行为、服务成本和长期影响；它适合上线前筛查和 bad-case 定位。
